# Provider Contracts

OpenLakeForge keeps the local stack as the only runnable environment today, but
platform modules should exchange provider-neutral contracts instead of assuming
one local implementation. A future cloud environment can then replace a contract
implementation without changing every consumer.

This is a cloud-readiness boundary only. It does not add AWS resources,
Keycloak, Vault, Secrets Manager, remote Terraform state, or production
hardening services.

## Contract Source Of Truth

Terraform is the source of truth for provider contracts. The local platform root
normalizes explicit contract objects in
`infra/terraform/environments/local/contracts.tf` and validates them with
Terraform `check` blocks. Runtime scripts can read the exported
`provider_contracts` output and fall back to the same local defaults before the
stack is applied.

The current hardening is provider-first, not service-replacement-first. Dagster,
Trino, Superset, OpenMetadata, dbt-duckdb, and Floe remain the implemented v1
solution stack. Their runtime configuration should depend on provider contracts
for storage, catalog, secrets, artifacts, identity, and access.

## Phase Contract

OpenLakeForge has two deployment phases:

1. Cluster foundation creates or selects the Kubernetes cluster.
2. Platform apply deploys the lakehouse services into that cluster with
   Terraform, Helm, and Kubernetes resources.

Local implements the cluster foundation with `kind` through the Terraform root
`infra/terraform/foundations/local-kind`, exposed as `make local-foundation-up`.
The local platform apply then uses the resulting `kind-openlakeforge-local`
kubeconfig context through `make local-infra-up`.

A future AWS implementation should keep the same split. The AWS cluster
foundation would create the VPC, subnets, routes, EKS control plane, node
groups, and cluster add-ons before the platform phase runs. This ordering is
required because EKS worker nodes run in VPC subnets and, with the usual AWS VPC
CNI model, pods consume VPC-routable IPs from those subnet ranges.

## Stable Contracts

| Contract | Local implementation now | Future cloud implementation shape |
| --- | --- | --- |
| Foundation | `foundation.kind` local Terraform root | `foundation.eks` root with VPC, subnets, EKS, node groups, add-ons |
| Kubernetes platform | `platform.kubernetes.kind` | Provider-managed Kubernetes API and workload identity |
| Storage | `storage.s3_compatible.seaweedfs` | `storage.aws_s3` or another S3-compatible object storage contract |
| Metadata database | `metadata_database.postgresql.in_cluster` | `metadata_database.aws_rds_postgresql` or another PostgreSQL-compatible contract |
| Catalog | `catalog.iceberg_rest.polaris` through `catalog_type = "rest"` | `catalog.aws_glue` through `catalog_type = "glue"` |
| Query | Trino Helm release | Same Trino contract, provider-specific resources and access |
| Reporting | Superset Helm release and local report PVC | Same Superset contract, externalized metadata/cache as needed |
| Orchestration | Dagster Helm release and project-code image loaded into kind | Same Dagster contract, registry-published images and cloud artifact storage |
| Artifacts | `artifacts.local_kind_and_s3` | `artifacts.ecr_and_s3` |
| Secrets | `secrets.kubernetes_secret` generated by Terraform | `secrets.aws_secrets_manager_or_external_secrets` |
| Identity | Local/basic app credentials | OIDC-capable identity contract, Keycloak or another provider later |
| Access | `kubectl port-forward` | Ingress, load balancer, private DNS, or another provider access contract |
| Observability | `observability.none` | Managed metrics/logging/tracing adapter later |

Consumers should depend on fields such as endpoint, bucket, region, Secret name,
database host, service name, image reference, access mode, and
`catalog_type`. They should not depend on whether the provider behind those
fields is SeaweedFS, S3, in-cluster PostgreSQL, RDS, kind, EKS, Polaris, or
Glue.

Product-owned runtime assets should use logical aliases. The local adapter
resolves `lakehouse_bronze` and `lakehouse_silver` to their respective
SeaweedFS-backed medallion buckets, and `iceberg_catalog` to Polaris. A future
cloud adapter can map the same logical names to S3 or ADLS prefixes without
changing any domain contract.

## Catalog Contract

The catalog contract describes an Iceberg catalog implementation. The local
provider sets:

```text
catalog_type = "rest"
catalog_provider = "polaris"
runtime_profile = "polaris-rest"
```

Polaris-specific fields such as REST URI, token URI, OAuth scope, and service
principal Secret names remain part of the local contract because local Floe,
dbt-duckdb, Trino, and OpenMetadata use them.

A future AWS provider can set:

```text
catalog_type = "glue"
catalog_provider = "aws-glue"
runtime_profile = "aws-glue"
```

That future implementation should not have to expose Polaris REST or OAuth
fields. Consumers that support more than one Iceberg catalog implementation must
branch on `catalog_type`.

## Local Defaults

The local provider profile remains intentionally lightweight:

- kind is the cluster foundation, managed by Terraform in a separate local
  foundation root.
- SeaweedFS implements the object storage contract.
- PostgreSQL runs in the cluster for Dagster, OpenMetadata, and Superset
  metadata.
- Kubernetes Secrets are the secret delivery mechanism.
- Basic/local application credentials are development-only.
- `kubectl port-forward` is the access model.
- Terraform state remains local and may contain development credentials.

These defaults are not production controls. They are the local implementation of
the same contracts a future provider profile should satisfy.

## Future AWS Notes

Do not add AWS resources in the current refactor. When AWS is implemented later,
use separate Terraform roots for the cluster foundation and platform apply:

- the AWS cluster root creates VPC, subnets, routing, EKS, node groups, and
  required add-ons, mirroring the local foundation root boundary;
- the AWS platform root consumes the EKS kubeconfig/cluster outputs and deploys
  the OpenLakeForge service modules;
- AWS storage, database, registry, secrets, identity, access, and catalog
  modules must produce the same contracts consumed by Trino, Dagster, Superset,
  and OpenMetadata.

This keeps the current local workflow intact while making the next provider
implementation a contract swap instead of a platform rewrite.
